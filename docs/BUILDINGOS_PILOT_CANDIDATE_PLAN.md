# BuildingOS Pilot Candidate Plan

## Record Status

`DOCUMENTATION_PLAN_V0_1_CANDIDATES_ONLY_NO_LIVE_PILOT_AUTHORIZATION`

## Prepared Date

2026-07-14

## Purpose

Propose a bounded set of small, medium, and large or complex BuildingOS pilot candidates for future Chief Architect review while preserving the frozen Foundation and Core, avoiding live-project activation, and separating public learning cases from authorised operational pilots.

This plan does not select a pilot, appoint a project team, obtain project authority, authorize implementation code, create a database or interface, connect external systems, or provide legal, certification, engineering-signoff, safety, contractual, planning, or other professional advice.

## Governing Principle

```text
A candidate tests the architecture.
A selected pilot tests an authorised implementation.
These are not the same decision.
```

## Candidate Status Vocabulary

### `CONCEPT_CANDIDATE`

A possible case identified for planning. No project permission, data access, implementation, or commitment exists.

### `DOCUMENTATION_LEARNING_CASE`

A case used only to test documents, classification, lifecycle, evidence, and interface architecture using fictional or approved public information.

### `READY_FOR_FOUNDER_OR_CHIEF_ARCHITECT_SELECTION_REVIEW`

A candidate has sufficient governance, ownership, scope, data, risk, and resourcing information to support a formal selection decision.

### `SELECTED_PILOT_NOT_ACTIVATED`

A Chief Architect selection decision exists, but implementation and live-project activation remain separately gated.

### `LIVE_PILOT_AUTHORIZED`

A separate explicit authorization defines scope, authority, data, implementation, security, professional review, and stop conditions.

No candidate in this plan has either of the final two statuses.

## Pilot Scale Classes

The plan uses BuildingOS internal labels:

- `BOS-S` — small;
- `BOS-M` — medium;
- `BOS-LC` — large or complex.

These labels must not be confused with NCC building classes, planning categories, legal classifications, project approvals, risk categories, certification states, or professional opinions.

## Candidate Evaluation Criteria

Every candidate should be assessed against the following criteria before selection.

### 1. Authority and Ownership

- identifiable project or asset owner;
- authority to participate;
- authority to share relevant information;
- clear separation between BuildingOS review and external decision authority;
- no reliance on assumed relationships or public reporting alone.

### 2. Bounded Scope

- explicit included and excluded scope;
- project, asset, site, package, or system boundary;
- manageable demonstration objective;
- clear stop conditions.

### 3. Evidence Availability

- lawful access to sufficient evidence;
- provenance and version visibility;
- ability to use fictional or approved public material where live data is unavailable;
- no dependency on copied standards, protected legal databases, or confidential project records.

### 4. Lifecycle Value

- enough stages, gates, reviews, decisions, changes, and handoffs to test the Lifecycle Stage Graph;
- suitable complexity for the target scale;
- visible re-entry, adaptation, commissioning, handover, or operational conditions where relevant.

### 5. Human Interface Value

- ability to test Portfolio Console, Project Cockpit, Stage Workspace, and Decision and Evidence Room;
- useful evidence, claim, review, decision, and ledger relationships;
- information density appropriate to the scale class.

### 6. ClimateOS Evidence Value

- meaningful site, climate-risk, environmental, building, MEP, BIM, construction, commissioning, operations, source, or provenance evidence;
- no transfer of ClimateOS conclusions or decision authority;
- ability to expose model uncertainty and source limits.

### 7. Regulatory Knowledge Value

- useful NCC, referenced-standard, NSW legislation or regulation, planning, approval, authority-guidance, jurisdiction, version, effective-date, applicability-claim, or human-review questions;
- no legal or certification conclusion required from BuildingOS;
- official sources remain authoritative.

### 8. Security and Publication Suitability

- public, private, licensed, confidential, personal, and security-sensitive boundaries can be defined;
- public repository content can remain safe;
- critical-infrastructure or protected detail is not required for the learning objective.

### 9. Resourcing and Duration

- realistic review effort;
- identifiable human contributors;
- bounded duration;
- no dependency on unapproved production infrastructure.

### 10. Reusability

- lessons can improve BuildingOS standards, lifecycle patterns, evidence structures, interface design, and documentation without overfitting the platform to one project.

## Candidate Progression Model

Every candidate should progress through separate gates:

```text
Candidate Identification
→ Documentation Learning Case
→ Candidate Review
→ Chief Architect Selection Decision
→ Pilot Architecture and Authority Brief
→ Explicit Implementation Authorization
→ Controlled Pilot Activation
→ Pilot Review and Closure
```

No later gate may be inferred from an earlier one.

# BOS-S Candidate Set

## S-01 — Tumut Small Building Adaptation and Secondary-Dwelling Learning Case

### Candidate Status

```text
CONCEPT_CANDIDATE
NOT_SELECTED
NOT_AUTHORIZED_AS_LIVE_PILOT
```

### Candidate Intent

Use a small residential adaptation, energy-resilience improvement, accessibility improvement, landscape-water intervention, secondary dwelling, or bounded building upgrade in Tumut as a possible `BOS-S` case.

The case may be real only after owner consent and a separate authority, privacy, data, and implementation decision. Before that point, it should be represented fictionally or through owner-approved non-sensitive information.

### Architecture Questions

- Can Project Genome remain compact while recording site, building, user, jurisdiction, climate, cost, design, construction, and operational boundaries?
- Can a short lifecycle graph preserve design, approval, procurement, construction, inspection, handover, and adaptation gates?
- Can BuildingOS make evidence and unresolved questions visible without turning a small project into bureaucracy?
- Can the interface remain readable for an owner while retaining professional and statutory boundaries?

### Potential Evidence Families

- owner brief and scope;
- site and climate context;
- measured or observed building conditions;
- sketches, drawings, BIM views, schedules, and specifications where authorised;
- product and fixture evidence;
- planning and approval metadata;
- construction photos and inspection records where lawful;
- handover and maintenance information;
- energy, water, comfort, or resilience observations;
- source and provenance records.

### Regulatory Knowledge Opportunities

- NCC edition and volume context;
- NSW and local planning or approval pathway metadata;
- referenced-standard identifiers and licensing notes;
- project-specific applicability claims requiring qualified human review;
- source-version and effective-date tracking.

### Required Boundaries

- owner consent;
- privacy and address controls;
- no personal financial information in the public repository;
- no claim that BuildingOS provides design, planning, certification, or legal advice;
- no automated approval or lifecycle transition.

### Provisional Scale and Governance

```text
Operating scale hypothesis: BOS-S
Governance depth hypothesis: GD-1 or GD-2
Official classification: not determined by BuildingOS
```

## S-02 — Fictional Small Community Asset Upgrade

### Candidate Status

`DOCUMENTATION_LEARNING_CASE`

### Candidate Intent

Use a fully fictional small hall, shelter, amenities building, landscape, water, or energy upgrade to test a compact public-asset case without personal data or project authority.

### Value

### Limitation

## Recommended Demonstration Sequence

The safest future demonstration sequence is:

1. fictional `BOS-S` fixture;
2. fictional `BOS-M` fixture;
3. S7 public-information `BOS-LC` structure-learning case;
4. review architecture and interface lessons;
5. only then consider whether a real small or medium project should be proposed for Chief Architect selection.

This is a planning recommendation, not implementation authorization or pilot selection.

## Pilot Selection Gate

Before selecting any real pilot, prepare a Candidate Selection Record addressing:

- exact candidate and bounded scope;
- owner and participation authority;
- evidence-access authority;
- data classification and publication boundary;
- project status and authoritative sources;
- Project Genome hypothesis;
- Lifecycle Stage Graph scope;
- intended Human Interface evaluation;
- ClimateOS evidence boundary;
- Regulatory Knowledge Layer scope;
- qualified human review requirements;
- legal, certification, contractual, planning, engineering, safety, and professional boundaries;
- implementation scope, if any;
- resource and duration estimate;
- stop conditions;
- closure and data-retention plan;
- Chief Architect decision required.

## Implementation Gate

Pilot selection does not authorize implementation.

Any implementation brief must separately define:

- approved code or design scope;
- repository and branch;
- architecture conformance;
- allowed data;
- fictional versus live records;
- security and privacy controls;
- tests and review evidence;
- prohibited capabilities;
- rollback and closure;
- explicit Chief Architect implementation authorization.

## Regulatory Knowledge Layer Boundary

Candidate work may identify and organise metadata for:

- Australian NCC sources, editions, volumes, amendments, adoption, commencement, effective dates, and jurisdiction context;
- referenced standards identifiers, versions, relationships, access, and licensing boundaries;
- NSW legislation, regulations, environmental planning instruments, approval records, authority guidance, and version status;
- Commonwealth, state, territory, and local variations;
- source change and supersession;
- applicability and compliance-related claims;
- human review and re-review triggers.

Candidate work must not:

- determine legal applicability;
- certify compliance;
- issue legal or professional advice;
- duplicate authoritative legal or standards content;
- treat a source summary as authoritative text;
- silently update project claims when a source changes.

## Knowledge Garden Boundary

Pilot-candidate documentation should remain compatible with the Markdown-first Knowledge Garden model:

- GitHub `main` is the approved source of truth;
- stable Markdown filenames and record IDs;
- ordinary links, backlinks, indexes, and maps of content;
- reviewable draft promotion;
- private research separated from approved records;
- controlled synchronisation;
- no blind two-way sync;
- no shadow copying of NCC, standards, legislation, paid databases, confidential project information, or security-sensitive content.

## Stop Conditions

Candidate development must pause for Chief Architect review before:

- selecting a real pilot;
- representing a candidate as an approved project participant;
- using non-public project data;
- changing BuildingOS product scope or architecture boundaries;
- asserting legal, certification, contractual, statutory, engineering-signoff, safety, or professional authority;
- modifying frozen Foundation or Core;
- authorizing implementation code;
- introducing a database, API, permissions, workflow, connector, runtime, production UI, application migration, or autonomous approval;
- activating ClimateOS transfer;
- using S7 as a live BuildingOS project.
