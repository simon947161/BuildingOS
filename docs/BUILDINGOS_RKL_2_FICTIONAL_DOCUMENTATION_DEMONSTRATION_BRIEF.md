## Purpose

Define a bounded, fictional, Markdown-first RKL-2 documentation demonstration that tests whether the RKL-1 source-governance architecture can be expressed as understandable, linked, reviewable project records without using real legal content, real project authority, implementation code, or automated regulatory capability.

RKL-2 is a documentation demonstration. It is not a legal knowledge service, compliance engine, source connector, database, certification system, or live project pilot.

## Governing Principle

```text
Demonstrate the governance structure without pretending to provide the law.
Demonstrate claims without pretending to decide applicability or compliance.
Demonstrate human review without pretending to transfer professional authority.
```

## Proposed Record Set

The future fictional demonstration record set should include the following linked records.

### 1. Fictional NCC Source Metadata Record

Demonstrate fields for:

- stable record ID;
- source category;
- fictional edition and volume context;
- fictional publication, adoption, commencement, transition, and effective dates;
- fictional jurisdiction context;
- source status;
- public or restricted locator boundary;
- metadata-only storage policy;
- supersession links;
- attributed review status.

The record must not reproduce, paraphrase, or imitate actual NCC provisions in a way that could be mistaken for authoritative content.

### 2. Fictional Referenced-Standard Metadata Record

Demonstrate:

- fictional standard identifier and title;
- fictional issuing body;
- fictional edition and amendment;
- fictional relationship to the demonstration source;
- access boundary;
- licensing note;
- metadata-only or link-only storage policy;
- supersession and human-review status.

No real paid-standard text or clause content may be copied.

### 3. Fictional NSW Instrument Metadata Record

Demonstrate:

- fictional Act or regulation category;
- fictional title and identifier;
- fictional publication, commencement, effective, transition, and repeal context;
- fictional geographic scope;
- fictional official publisher class;
- fictional source locator;
- review status and limitations.

The record must not use a real legislative title or provision unless it is clearly referenced only as a category label and contains no legal interpretation or copied content.

### 4. Fictional Jurisdictional Variation Record

Demonstrate a relationship between:

- a fictional national base source;
- a fictional NSW adoption or variation context;
- an affected fictional topic;
- fictional effective dates;
- assumptions and uncertainty;
- human-review requirement;
- supersession trigger.

The record must state that it does not determine the legal effect of any real jurisdictional variation.

### 5. Fictional Regulatory Provision Reference

Demonstrate a bounded locator to a fictional section, clause, schedule, performance requirement, approval condition, or standard clause.

It must include:

- source version and date context;
- project subject;
- no-copy note;
- limitations;
- review status.

### 6. Fictional Regulatory Relationship Record

Demonstrate relationships such as:

- fictional code references fictional standard;

- fictional jurisdiction varies fictional base source;

Link a fictional source or provision to:

- Project Genome `DEMO-RKL2-001`;

This record identifies a review need. It does not determine applicability.

Required fields:

Demonstrate a bounded statement about the relationship between a fictional project state and a fictional requirement.

Required fields:

- exact fictional subject assessed;

- fictional evidence reviewed;
- review basis;
- exclusions and outstanding items;
- reviewer identity and attributed fictional capacity;
- validity context;
- re-review trigger;
- external certification or approval required.

Mandatory statement:

```text
This fictional compliance-related claim is not a certificate, consent, approval, professional sign-off, or general conclusion.
```

### 10. Fictional Human Review Record

Demonstrate:

- fictional reviewer identity;
- fictional role or capacity;
- explicit statement that the role is fictional;

## Markdown and Frontmatter Rules

Records should use portable Markdown and simple YAML frontmatter.

Required common fields:

```yaml
record_id: stable-fictional-id
record_type: fictional-record-type
title: fictional-title
demonstration_only: true
fictional_record: true
legal_effect: none
project_authority: none
legal_advice: false
certification: false
human_review_required: true
status: draft|in-review|reviewed|superseded|closed
created_on: YYYY-MM-DD
updated_on: YYYY-MM-DD
related_records: []
limitations: []
```

Obsidian-specific features may be used only as optional enhancements. The records must remain readable through ordinary Markdown tools.

## Human Review Demonstration

The record set should demonstrate at least three different fictional review outcomes:

1. reviewed with limitations;
2. requires further evidence or external confirmation;
3. superseded and sent for re-review.

Each review must show:

- who reviewed;
- fictional attributed capacity;
- what was reviewed;
- evidence considered;
- assumptions;
- limitations;
- what remains unresolved;
- decision-authority boundary;
- re-review trigger.

No review may claim actual professional competence, statutory appointment, certification status, or legal authority.

## Demonstration Scenarios

The future record set should demonstrate these linked scenarios:

### Scenario A — Version and Effective-Date Context

A fictional source has been published but is not yet effective. The project reference and applicability claim must preserve both publication and effective-date context.

### Scenario B — Referenced Standard and Access Boundary

A fictional code references a fictional licensed standard. The repository stores metadata and an access note only, not the standard content.

### Scenario C — NSW Variation and Human Review

A fictional NSW variation may affect a fictional project subject. The record remains an applicability claim pending human confirmation.

### Scenario D — Compliance-Related Claim With Outstanding Evidence

A fictional design state appears related to a fictional requirement, but inspection or commissioning evidence is missing. The claim remains limited and cannot be represented as certification.

### Scenario E — Source Change and Supersession

A fictional amendment supersedes a prior source version. Existing claims remain preserved and are flagged for human re-review rather than silently rewritten.

## Stop Conditions

Stop and return for review if:

- a record could be mistaken for a real legal, NCC, standards, approval, certification, or project record;
- real legal text, paid-standard content, confidential project information, or protected data appears necessary;
- an applicability claim begins to be treated as a legal determination;
- a compliance-related claim begins to be treated as certification or approval;
- reviewer access is treated as competence or authority;
- a source change silently alters a prior claim or decision;
- executable workflow, permissions, automatic transition, connector, scraper, database, API, runtime, UI, or code becomes necessary;
- frozen Foundation or Core changes appear necessary;
- a live pilot or real project is introduced.
