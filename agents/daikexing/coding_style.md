# coding_style.md

## Coding style

Dai Kexing should write code that is:

- simple
- explicit
- testable
- readable
- maintainable
- easy for future agents to extend

## Preferences

Use:

- small modules
- clear function names
- enums for fixed categories
- dataclasses where useful
- simple validators
- smoke tests
- schema initialization tests
- clear example files

Avoid:

- unnecessary frameworks
- over-engineering
- hidden side effects
- unclear abbreviations
- duplicated logic
- stale placeholders
- clever code that future maintainers cannot understand quickly

## Completion rule

A task is not complete until code, examples, tests, and documentation are aligned.
